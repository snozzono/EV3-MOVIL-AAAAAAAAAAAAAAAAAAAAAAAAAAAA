Modelos (resumen):
- Product: id, name, description?, price?, stock, brand, category, images?, enabled?
- ProductImage: path, name?, type?, size?, mime?, access?, url?, meta(width?, height?)
- Cart: id, createdAt, userId; CartItem: id, createdAt, quantity, cartId, productId
- Order: id, createdAt, total?, status?, userId; OrderItem: id, createdAt, quantity, price?, orderId, productId
- Shipping: id, createdAt, address, shippingDate?, status?, orderId
- User: id, name, email, role, status?, shippingAddress?, phone?, createdAt
- Requests/Responses: LoginRequest, AuthResponse, Create/Update Product, CreateProductResponse, Create/Update User, Cart/Order/Shipping Requests

Capas:
- UI: Activities/Fragments/Adapters con ViewBinding y RecyclerView.
- Estado: TokenManager (SharedPreferences), CartManager (cart_id persistido).
- Red: RetrofitClient + OkHttp (interceptor Authorization), Services REST.
- Utilidades: ImageUrlResolver (resuelve url vs path relativo).

Por qué funciona bien:
- Separación de responsabilidades clara; modelos consistentes; ViewBinding; ListAdapter + DiffUtil.
- Flujos didácticos y simples; manejo básico de estados.
- Resolver de imágenes robusto ante respuestas parciales.

Mejoras (xyz):
- MVVM + Repository + UseCases, StateFlow y manejo de errores unificado.
- Room para carrito/catálogo; soporte offline; caché.
- Hilt para DI; DataStore/EncryptedSharedPreferences para token.
- Compresión/transformación de imágenes; webp/avif; Paging 3 en listas.
- Tests de unidad e instrumentados básicos.

Guion de demo:
- Admin: login → CRUD productos (múltiples imágenes) → usuarios (bloqueo) → órdenes: aceptar y rechazar.
- Cliente: login → catálogo → carrito (editar cantidades) → pago simulado → estado → perfil.
- Cierre técnico: Retrofit, ViewBinding, RecyclerView, SharedPreferences, ImageUrlResolver.

Actualización — Definiciones clave:
- StateFlow: flujo “caliente” con valor actual para estado de UI.
- Room: BD local con entidades/DAOs; soporta Flow y se integra con Paging.
- Paging 3: paginación con `Pager` → `Flow<PagingData<T>>` y `PagingDataAdapter`.
- BuildConfig: constantes por variante definidas con `buildConfigField` y consumidas en código.

Respuestas del doble-check (estado actual):
- URLs/credenciales: configuradas en BuildConfig y usadas via ApiConfig; verificar grupos.
- Logout/sesión: `TokenManager.clear()` y navegación por rol correctas.
- CartManager: asegura `cart_id`; reinstalaciones crean nuevo (depende del backend).
- Imágenes: `ImageUrlResolver` cubre `url` y `path` relativo en vistas clave.
- Validación: básica; faltan formato de email y rangos (`price >= 0`, `stock >= 0`).
- Estados: hay `progress` y toasts; falta patrón unificado y reintento visible.
- Endpoints: alineados con modelos y rutas; confirmar esquemas exactos con backend.
- Imágenes: sin validación de tamaño/formatos; confirmar restricciones en servidor.
- Seeds: recomendable ≥10 productos y varios usuarios para demo fluida.

Backend a confirmar:
